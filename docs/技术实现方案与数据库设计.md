# 游戏工会管理平台 — 技术实现方案与数据库设计

---

## 一、技术架构建议

### 1.1 整体架构

采用前后端分离架构：

| 层级 | 技术选型建议 | 说明 |
|-----|------------|------|
| 前端 | Vue 3 + Element Plus / Ant Design Vue | 响应式设计，适配PC端 |
| 后端 | Spring Boot 3.x + MyBatis Plus | Java生态，成熟稳定|
| 数据库 | MySQL 8.0 + Redis | 主数据存储+缓存 |
| 消息队列 | RabbitMQ / Redis Stream | 异步通知、任务调度 |
| 文件存储 | MinIO / 阿里云OSS | 头像、宣传图存储 |
| 定时任务 | XXL-Job / Spring Scheduler | 任务状态自动流转 |

### 1.2 核心设计原则

1. **数据隔离优先**：所有业务表必须包含 `game_partition_id` 字段，查询时强制带分区条件
2. **权限双重校验**：Controller层校验用户权限，Service层校验数据归属分区+工会
3. **操作留痕**：敏感操作异步写入操作日志表，记录不可删除
4. **高并发处理**：接单、申请等操作使用数据库乐观锁或分布式锁

### 2.2 关键业务逻辑实现

#### 任务状态自动流转（定时任务）

```sql
-- 每分钟执行：检查任务状态
-- 1. 未开始 -> 进行中（到达开始时间）
UPDATE guild_task 
SET status = 2, updated_at = NOW()
WHERE status = 1 AND start_time <= NOW();

-- 2. 进行中/已满员 -> 已结束/已作废（到达截止时间）
-- 参与人数 >= 下限 -> 已结束
UPDATE guild_task 
SET status = 4, updated_at = NOW()
WHERE status IN (2, 3) AND end_time <= NOW() AND current_players >= min_players;

-- 参与人数 < 下限 -> 已作废
UPDATE guild_task 
SET status = 5, updated_at = NOW()
WHERE status IN (2, 3) AND end_time <= NOW() AND current_players < min_players;
```

#### 接单并发控制（乐观锁）

```java
// 伪代码：接单操作
@Transactional
public boolean takeTask(Long taskId, Long userId) {
    // 1. 查询任务（带版本号）
    Task task = taskMapper.selectById(taskId);
    
    // 2. 校验任务状态
    if (task.getStatus() != 2) { // 非进行中
        throw new BizException("任务不可接单");
    }
    
    // 3. 校验是否已满员
    if (task.getCurrentPlayers() >= task.getMaxPlayers()) {
        throw new BizException("任务已满员");
    }
    
    // 4. 校验是否已接单
    if (participantMapper.exists(taskId, userId)) {
        throw new BizException("不可重复接单");
    }
    
    // 5. 更新任务人数（乐观锁）
    int affected = taskMapper.increasePlayerCount(taskId, task.getVersion());
    if (affected == 0) {
        throw new BizException("接单失败，请重试");
    }
    
    // 6. 添加参与记录
    participantMapper.insert(taskId, userId);
    
    // 7. 检查是否满员，更新状态
    task = taskMapper.selectById(taskId);
    if (task.getCurrentPlayers() >= task.getMaxPlayers()) {
        taskMapper.updateStatus(taskId, 3); // 已满员
    }
    
    return true;
}
```

#### 申请冷却检查

```sql
-- 提交申请前检查
SELECT cooldown_end_time FROM application_cooldown
WHERE guild_id = ? AND user_id = ? AND cooldown_end_time > NOW();

-- 拒绝申请后更新冷却记录
INSERT INTO application_cooldown (guild_id, user_id, reject_count, cooldown_end_time, last_reject_time)
VALUES (?, ?, 1, DATE_ADD(NOW(), INTERVAL 72 HOUR), NOW())
ON DUPLICATE KEY UPDATE
reject_count = CASE 
    WHEN last_reject_time < DATE_SUB(NOW(), INTERVAL 24 HOUR) THEN 1
    ELSE reject_count + 1
END,
cooldown_end_time = CASE
    WHEN reject_count + 1 >= 3 THEN DATE_ADD(NOW(), INTERVAL 72 HOUR)
    ELSE cooldown_end_time
END,
last_reject_time = NOW();
```

## 四、安全设计

### 4.1 数据隔离保障

1. **数据库层**：所有查询强制带 `game_partition_id` 条件
2. **Service层**：操作前校验数据归属的分区是否与用户当前分区一致
3. **接口层**：JWT Token中包含用户当前选择的分区ID，接口自动校验

### 4.2 敏感操作保护

| 操作 | 保护措施 |
|-----|---------|
| 解散工会 | 二次确认弹窗 + 输入"确认解散" + 校验无进行中任务 |
| 踢出成员 | 必须填写原因 + 弹窗确认 |
| 任免管理员 | 二次确认 + 推送通知给当事人 |
| 修改工会名 | 实时校验唯一性 + 操作日志留痕 |

### 4.3 防刷机制

| 场景 | 限制规则 |
|-----|---------|
| 发布任务 | 同一工会24小时最多10个 |
| 提交申请 | 冷却期内禁止重复提交 |
| 登录 | 错误5次锁定30分钟 |
