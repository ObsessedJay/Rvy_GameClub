### 1.2 单点登录方案

**方案：共享Token + 扩展用户信息**

若依系统已有完整的JWT Token认证机制，GameClub直接复用：

```
用户登录流程：
1. 用户通过若依登录接口认证
2. 生成JWT Token（包含userId, username, roles）
3. Token存储在Redis中（若依已实现）
4. GameClub接口携带同一Token访问
5. GameClub从Token解析用户信息，查询用户在各工会的身份
```

**Token扩展字段：**

```json
{
  "user_id": 1,
  "user_name": "admin",
  "roles": ["admin"],
  "current_partition_id": 1,  // 新增：当前游戏分区
  "current_guild_id": 100     // 新增：当前工会（可选）
}
```

## 二、需求文档问题诊断

### 2.1 严重问题（必须修复）

#### 问题1：游戏分区与若依部门的关系未定义

**问题描述：**
需求文档中设计了独立的`game_partition`表，但未说明与若依`sys_dept`表的关系。

**解决方案：**
两种方案可选：

| 方案 | 说明 | 优缺点 |
|-----|------|--------|
| 方案A：复用sys_dept | 游戏分区作为若依部门的子级 | ✅复用现有权限体系 ❌部门层级混乱 |
| 方案B：独立表 | 新建game_partition表 | ✅数据隔离清晰 ❌需独立权限控制 |

**推荐方案B**，理由：
- 游戏分区是业务概念，不是组织架构
- 工会管理需要独立的权限模型
- 避免污染若依原有的部门体系

#### 问题2：工会角色与若依角色的冲突

**问题描述：**
- 若依角色：超级管理员、普通角色（系统级）
- 工会角色：会长、管理员、普通成员（业务级）
- 两套角色体系如何共存？

**解决方案：**
工会角色不使用若依的sys_role，而是独立设计：

```sql
-- 工会成员表中的role字段
-- 1=会长, 2=管理员, 3=普通成员
-- 这是业务角色，与若依系统角色无关
```

**权限校验流程：**

```
请求 → 若依权限校验（是否有菜单权限）→ GameClub权限校验（是否是会长/管理员）
```

#### 问题3：数据隔离实现细节缺失

**问题描述：**
需求强调"游戏分区数据隔离"，但未给出具体实现方案。

**解决方案：**
使用MyBatis拦截器实现自动隔离：

```java
@Intercepts({@Signature(type= Executor.class, method="query", args={...})})
public class PartitionInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) {
        // 自动为所有查询添加 game_partition_id 条件
        // 从ThreadLocal获取当前用户的分区ID
    }
}
```

## 五、权限校验设计

### 5.1 双重权限校验

```
请求 → 若依菜单权限 → GameClub业务权限 → 执行业务
```

**若依菜单权限：** 校验用户是否有访问GameClub菜单的权限

**GameClub业务权限：** 校验用户在当前工会的角色

### 5.2 权限注解设计

```java
// 自定义注解：工会权限
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface GuildRole {
    int[] value(); // 1=会长, 2=管理员, 3=普通成员
}

// 使用示例
@GuildRole({1, 2}) // 会长或管理员
@PostMapping("/task/create")
public Result createTask(@RequestBody TaskDTO dto) {
    // ...
}
```

### 5.3 权限拦截器

```java
@Component
public class GuildRoleInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        // 1. 获取方法上的@GuildRole注解
        // 2. 获取当前用户在当前工会的角色
        // 3. 校验是否满足注解要求
    }
}
```

## 八、风险与应对

| 风险 | 级别 | 应对方案 |
|-----|------|---------|
| 高并发接单冲突 | 高 | 乐观锁 + Redis分布式锁 |
| 跨分区数据越权 | 高 | MyBatis拦截器 + 双重校验 |
| 权限体系混乱 | 中 | 明确区分系统角色和业务角色 |
| 任务状态流转延迟 | 中 | Quartz定时任务 + 消息队列补偿 |
| 数据库表过多 | 低 | 合理设计，避免冗余 |
